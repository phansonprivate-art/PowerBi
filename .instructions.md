# POWER BI PBIP Automation Framework

**Version:** 1.0.0 | **Kompatibilität:** PBI Desktop March 2026+ | **Format:** PBIR Enhanced  
**Repository:** https://github.com/phansonprivate-art/PowerBi

---

## Kontext: Worum geht es?

Dieses Framework automatisiert die Erstellung von Power BI PBIP Projekten in 3 Phasen:

- **Phase 1**: Anforderungsanalyse via MCP (Tabellen, Measures, Relationships)
- **Phase 2**: Technische Umsetzung (TMDL Export, PBIP Dateistruktur, visual.json)
- **Phase 3**: Validierung (8 Automated Checks vor PBI Desktop öffnen)

### ⚠️ Kritische Regel
**TMDL = Einzige Feldquelle!**
- MCP liefert Feldnamen des aktuellen Modells
- Das ZIEL-Modell kann ANDERE Feldnamen haben
- → visual.json Feldnamen IMMER aus TMDL-Dateien, NIEMALS von MCP erfinden!

## Schnell-Workflow

### Phase 1: Analyse
```
1. parameters.md ausfüllen (PROJEKTNAME, ZIELORDNER)
   → 09_TEMPLATE ist fest: C:\Users\phant\.claude\PowerBi\09_TEMPLATE
2. MCP Verbindung testen (connection_operations, database_operations)
3. Tabellen + Measures inventarisieren
4. Use Cases + Anforderungen dokumentieren
```

### Phase 2: Umsetzung
```
1. Measures via MCP erstellen (measure_operations, displayFolders)
2. TMDL exportieren (database_operations → ExportToTmdlFolder)
3. TMDL-Feldliste extrahieren (python extract-tmdl-fields.py)
4. PBIP Struktur erstellen (definition.pbism, definition.pbir, .pbip)
5. page.json + visual.json generieren (AUS TMDL-FELDLISTE!)
```

### Phase 3: Validierung
```
CHECK 0: TMDL-Feldliste vorhanden? (GATE)
CHECK 1: Semantic Model Struktur OK?
CHECK 2: PBIP ROOT Struktur OK?
CHECK 3: JSON Syntax (python validate-json-syntax.py)
CHECK 4: Schema-Versionen (4.2, 2.0.0, etc.)
CHECK 5: Verbotene Felder (kein "Source", kein "filterConfig")
CHECK 6: Datei-Vollständigkeit (pages.json, visual.json)
CHECK 7: TMDL Feldvalidierung (python validate-visuals.py) ← KRITISCH!
CHECK 8: Daten-Integrität (MCP Measures/Relationships)
```

## Die 11 Kritischen Regeln (NIEMALS brechen!)

| # | Regel | Richtig | Falsch |
|---|-------|---------|--------|
| 1 | **TMDL = Feldquelle** | Aus TMDL lesen | Von MCP erfinden |
| 2 | **definition.pbism version** | `"4.2"` | `"1.0.0"` oder `"1"` |
| 3 | **.pbip artifacts** | NUR `{"report": {...}}` | Mit `"semanticModel"` |
| 4 | **visual.json SourceRef** | `"Entity": "..."` | `"Source": "..."` |
| 5 | **CHECK 7 vor PBI Desktop** | Feldnamen validieren | Direkt öffnen = Fehler |
| 6 | **Entity für Measures** | `"Measure"` | `"_Measures"` |
| 7 | **Property = TMDL-Feldname** | Exakter Name aus TMDL | MCP-Name oder erfunden |
| 8 | **query innerhalb visual{}** | `visual.query.queryState` | Top-Level `query` |
| 9 | **queryState = Object** | `{}` mit Rollen-Keys | `[]` mit SetBindingRef |
| 10 | **Measures in TMDL** | In Measure.tmdl eintragen | Nur MCP (In-Memory) |
| 11 | **Template max 2 Seiten** | 3. Seite manuell kopieren | Mehr als 2 im Script |

```powershell
python3 validate-json-syntax.py --pbip-path "C:\...\Sons_Final_Result"
python3 validate-visuals.py --pbip-path "C:\...\Sons_Final_Result" --tmdl-list "TMDL-Feldliste.json"
```

## Workflow-Tipps

### ✅ DoS
- Schreibe Phase 1-3 der Reihe nach
- CHECK 0-8 IMMER komplett durchlaufen
- TMDL-Feldliste als Heilige Schrift behandeln
- Bei Unsicherheit: CRITICAL_RULES.md lesen

### ❌ Don'ts
- Feldnamen von MCP in visual.json verwenden
- version in definition.pbism anders als "4.2" setzen
- "semanticModel" in .pbip artifacts einfügen
- "Source" statt "Entity" in SourceRef nutzen
- PBI Desktop öffnen vor CHECK 7 PASS
